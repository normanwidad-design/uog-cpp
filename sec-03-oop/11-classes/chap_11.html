Member access

✎  Each member of a class type (struct or class) have a property called an access level.

✎  Access level determines who can access that member:

  1. public
  2. private
  3. protected

Members of a struct are public by default

Terminologies

  • Members that have the public access level are called public members.
  • Public members are members of a class type that do not have any restrictions on how they can be accessed.

Meaning

  • Public members can be accessed by other members of the same class.
  • Public members can also be accessed by the public, which is what we call code that exists outside the members of a given class type.

Key points to remember

  • The members of a struct are public by default.
  • Public members can be accessed by other members of the class type, and by public

What is an aggregate?

  • In general programming, an aggregate data type (also called an aggregate) is any type that can contain multiple data members.
  • Some types of aggregates allow members to have different types (e.g. structs).
  • Others require that all members must be of a single type (e.g. arrays).

We now learn constructors in C++.

Intro to constructors: Rationale

➥  We have just learned we can do this:

struct Foo // Foo is an aggregate
{
    int x {};
    int y {};
};

int main()
{
    Foo foo { 6, 7 }; // uses aggregate initialization
    return 0;
}

Intro to constructors: Rationale

But we cannot do this:

class Foo // Foo is not an aggregate (has private members)
{
    int x {};
    int y {};
};
int main()
{
    Foo foo { 6, 7 }; // compile error: can not use aggregate initialization
    return 0;
}

Not allowing class types with private members to be initialized via aggregate initialization makes sense:

  • Aggregate initialization requires knowing about the implementation of the class, which we’re intentionally trying to avoid when we hide our data members.
  • If our class had some kind of invariant (e.g., \(18 < \text{age} < 65\) for working age), we would be relying on the user to initialize the class in a way that preserves the invariant.

➜  That’s why we need a constructor and it can do all the security check if needed. – just like you learned in Python.

Constructors

A constructor is a special member function that is automatically called after a non-aggregate class type object is created.

When a non-aggregate class type object is defined, the compiler looks to see if it can find an accessible constructor that is a match for the initialization values provided by the callers.1

Key insight

  • Many new programmers are confused about whether constructors create the objects or not.
  • They do not – the compiler sets up the memory allocation for the object prior to the constructor call. The constructor is then called on the uninitialized object.
  • After the constructor finishes executing, we say the object has been “constructed”, and the object should now be in a consistent, usable state.

Note

✎  Aggregates are not allowed to have constructors – so if we add a constructor to an aggregate, it is no longer an aggregate.

Naming constructors

  • Constructors must have the same name as the class (with the same capitalization).
  • For template classes, this name excludes the template parameters.2
  • Constructors have no return type (not even void)

Member initialization via a member initialization list

✎  Modern C++ provides a mechanism even better than writing this->variable = variable in the constructor body: member initializer list.

Member initialization via a member initialization list

Member initializer list

  • is defined after the constructor parameters.
  • begins with a colon :
  • lists each member to initialize along with the initialization value for that variable, separated by comma
  • uses a direct form of initialization – preferably braces, but parentheses work as well

Example (again)

class Vector2D
{
    double x {}, 
    double y {};
public:
    Vector2D(double x, double y): x { x }, y { y } {}
};

Member initialization list: why better than this->x = x

✎  Member Initalizer List: x { x }

Why it’s better:

  • initialization, not assignment: created and given value in one single step
  • required for const and references: if class has const members or reference variables (e.g. double& ref), we must use this method. These teypes cannot be left unintialized and assigned to later.
  • performance: avoid calling default constructor followed by an assignment operator, goes straight to the copy/move constructor (discussed later)

✎  Assignment in the Body: this->x = x

Sequence of events (you see it is worse than above)

  • implicit initialization: initializes x with garbage value (if defined double x), or to default value (if defined double x { })
  • execution: constructor body starts
  • assignment: value of the parameter x is copied into this->.

: x { x} versus this->x = x: Key differences

Feature Member Initialization List : x { x } Body Assignment this->x = x
Action direct initialization Default init + assignment
Efficiency Faster (no double-work) Slower for complex objects
Const/Refs Works perfectly Fails to compile
Ordering Follows declaration order in class Follows code order in body

Best practice

Prefer using the member initializer list to initialize your members over assigning values in the body of the constructor.

Member initializer list formatting – Note C++ does not care space

  • One line
Foo(int x, int y) : x { x }, y { y }
{
}
  • Multiple lines with colon : after class name
Foo(int x, int y) :
    x { x },
    y { y }
{
}
  • Multiple lines with colon : before the first initializer
Foo(int x, int y)
    : x { x }
    , y { y }
{
}

✎  If member initialization list is trivial, all initializers can go one line. Otherwise, third style is preferable

Member initializer list vs default member initializers

Members are initialized in a few different ways:

  • If a member is listed in the member initializer list, that initialization value is used.
  • Otherwise, if the member has a default member initializer, that initialization value is used.
  • Otherwise, the member is default-initialized.

Constructor function bodies

  • The bodies of constructors functions are most often left empty – because we primarily use constructor for initialization, which is done via member initializer list (if that’s all we need to do).
  • We can add statements to constructor function bodies to do any other setup tasks required (e.g. open a file or databa, allocate memory, check invariant \(\ldots\))

Default constructors and default arguments

Default constructor

[Default constructor] is a constructor that accepts no arguments.

Value initialization vs default initialization for class types

✎  If a class type has a default constructor, both value initialization and default initialization will call the default constructor.

// With default constructor, value initialization and default initialization will
// call the default constructor.
Foo foo1{}; // value initialization, calls Foo() default constructor
Foo foo2;   // default initialization, calls Foo() default constructor

Contructors with default argument

✎  As with all functions, the rightmost parameters of constructors can have default arguments.

✎  If all the params in a constructor have default args, the constructor is a default constructor – as it can be called with no args.

Overloaded constructors

✎  Because constructors are functions, they can be overloaded.

#include <iostream>

class Foo
{
private:
    int x {};
    int y {};

public:
    Foo() // default constructor
    {
        std::cout << "Foo constructed\n";
    }

    Foo(int x=1, int y=2) // default constructor
        : x { x }, y { y }
    {
        std::cout << "Foo(" << x << ", " << y << ") constructed\n";
    }
};

int main()
{
    Foo foo{}; // compile error: ambiguous constructor function call
    return 0;
}

➜  The compiler will look for a default constructor & and will find two, leading to ambiguity.

An implicit default constructor

  • If a non-aggregate class type (e.g. private data members) object has no user-declared constructors, the compiler will generate a public default constructor.
  • This constructor is called an implicit default constructor

An implicit default constructor

✎  The implicit default constructor is equivalent to a constructor that has

  • no parameters
  • no member initializer list
  • no statements in the constructor body
public:
    Foo() // implicitly generated default constructor
    {
    }

Killing implicit default constructor

✎  If we define any other constructor which is not a default constructor, the compiler automatically deletes the implicit default constructor.

➥  Consequently, we need to write a default constructor if we need one. Instead of writing implicit default constructor “explicitly”, we can use = default.

Intro to the copy instructor

Copy constructor

✎  A copy constructor is a constructor that is used to initialize an object with an existing object of the same type.

✎  After the copy constructor executes, the newly created object should be a copy of the object passed in as the initializer.

An implicit copy constructor

✎  If we do not provide a copy constructor for our classes, C++ will create a public implicit copy constructor for you.

Copy constructor

✎  We can explicitly define our own copy constructor.

Copy constructor: Best practices

  • A copy constructor should not do anything other than copy an object.
  • This is because the compiler may optimize the copy constructor out in certain cases.

Best practice

Copy constructors should have no side effects beyond copying.

Best practice

Prefer the implicit copy constructor, unless you have a specific reason to create your own.

The copy constructor’s parameter must be a reference

  • It is a requirement that the parameter of a copy constructor must be an lvalue reference or const lvalue reference.
  • The copy constructor should not be modifying the parameter, using a const lvalue reference is preferred.

Best practice

If you write your own copy constructor, the parameter should be a const lvalue reference.
